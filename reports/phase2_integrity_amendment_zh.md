# Phase 2 正式实验完整性修订说明

冻结时间：2026-07-13T12:37:59.9482667Z

## 为什么需要这次修订

Phase 2 想回答的问题是：模型生成错误的“歌名-歌手”关系后，内部机制标签能否预测哪种自我纠错提示更有效。这个结论成立的前提是：进入机制实验的关系确实来自冻结的 Granite 生成批次，确实经过两套目录的严格核验，而且后续 GPU 结果没有混入别的协议、模型、bundle 或脚本。

独立代码审查发现，原实现的科学思路没有明显 blocker，但实验资产仍可能在中断恢复、目录截断、最终导出或分片合并时被误用。因此，在正式机制和纠错 GPU 实验之前，先冻结以下保守修订。

## 大白话解释

1. 搜索结果如果可能只返回了前 100 或 200 条，我们不再把“没在这一页看到模型写的歌手”当成它不存在，而是直接排除这条样本。
2. 如果 MusicBrainz 用模型写出的歌手名进行定向搜索时，返回了同一个正规歌手，我们把它视为“可能是别名”，不算严格错误。
3. 每一条最终样本都必须证明自己来自冻结协议、冻结模型版本和原始生成记录，并能追溯到完整的目录 HTTP 响应。
4. 评分前重新计算协议、目录文件、原始证据、verifier、bundle 和实际 GPU runner 的 SHA-256；任一文件被换过就拒绝评分。
5. GPU 实验按固定记录区间分片，每完成一小批就在 Job 日志中留下 checkpoint。任务中断后可以证明哪些记录完成了，也能精确列出缺失 ID。
6. 如果 `prior_masked` 或 `wrong_binding` 任一类少于预注册的 15 个标题，H2 直接记为“未检验”，不再计算一个看似正式的 bootstrap 置信区间。

## 防止看结果改规则

修订前只看过运行中的汇总计数；另外，为检查输入 provenance schema，终端显示过一条尚未核验的原始生成记录。没有查看任何逐条目录标签、目录响应、最终入选 cluster、机制分数或干预结果。

本次没有改模型、提示词、上下文、种子、采样参数、样本上限、标题聚类、哈希盐、诊断层、patching 方法、纠错提示或假设阈值。新增规则只会排除证据不足或疑似别名的旧候选，不会把旧版非冲突样本升级成冲突。

## 当前验证

- 全量单元测试：190 个通过，1 个按原设计跳过。
- 目录反例覆盖：结果窗口截断、别名疑点、首条请求中断、同 record ID 偷换关系。
- Runner 反例覆盖：分片范围、checkpoint 解码、来源不一致、冲突重复、缺失 ID、catalog hash 缺失。
- H2 边界覆盖：14/15、0/15 均停止且不 bootstrap；15/15 才允许检验。

## 下一步

1. 保留正在运行的旧版 primary 目录结果作为审计材料，不直接用于选样。
2. 用 v2 verifier 从已归档原始响应重放全部 290 条；只对缺失或失败请求联网重试。
3. 按原预注册门槛决定是否运行已经冻结的 20 个扩展歌单。
4. 合并 primary 与 extension 后，如果严格冲突标题少于 30，停止机制实验并公开样本产出边界。
5. 如果达到 30，冻结最终 diagnosis、candidate-free correction 和离线 scoring 三套物理隔离资产，再提交分片 GPU Jobs。
